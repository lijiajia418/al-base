  二、主数据模型                                                           
                                                                
  10 张表，3 层结构
                                                                           
  组织层:       schools
                │                                                          
  用户层:       users ─── teacher_profiles                      
                │    ─── student_profiles                                  
                │    ─── parent_profiles                                   
                │
  关系层:       classes                                                    
                ├── class_teachers    (老师↔班级, N:M)          
                ├── class_students    (学生↔班级, N:M)                     
                │                                                          
                parent_student_relations (家长↔学生, N:M)                  
                invitations              (邀请码)                          
                                                                           
  各表核心字段说明
                                                                           
  schools — 顶层租户                                            

  ┌──────────┬───────────────────────────────────────────┐                 
  │   字段   │                   说明                    │
  ├──────────┼───────────────────────────────────────────┤                 
  │ id       │ UUID 主键                                 │      
  ├──────────┼───────────────────────────────────────────┤
  │ name     │ 学校名称                                  │
  ├──────────┼───────────────────────────────────────────┤
  │ code     │ 学校识别码/邀请码，唯一                   │
  ├──────────┼───────────────────────────────────────────┤                 
  │ settings │ JSONB，学校级配置（评分口径、提醒规则等） │
  └──────────┴───────────────────────────────────────────┘                 
                                                                
  users — 统一用户表（所有人一张表）                                       
                                                                
  ┌───────────────┬─────────────────────────────────────────────┐          
  │     字段      │                    说明                     │
  ├───────────────┼─────────────────────────────────────────────┤
  │ id            │ UUID 主键                                   │
  ├───────────────┼─────────────────────────────────────────────┤
  │ school_id     │ FK → schools，可为 NULL（家庭模式独立用户） │          
  ├───────────────┼─────────────────────────────────────────────┤          
  │ email         │ 登录邮箱，唯一                              │          
  ├───────────────┼─────────────────────────────────────────────┤          
  │ phone         │ 手机号                                      │
  ├───────────────┼─────────────────────────────────────────────┤          
  │ password_hash │ 密码哈希                                    │
  ├───────────────┼─────────────────────────────────────────────┤          
  │ name          │ 姓名                                        │
  ├───────────────┼─────────────────────────────────────────────┤
  │ roles         │ TEXT[]，角色数组                            │
  ├───────────────┼─────────────────────────────────────────────┤
  │ status        │ active / inactive / suspended               │
  └───────────────┴─────────────────────────────────────────────┘          
   
  为什么统一一张表：                                                       
  - 避免三张用户表的 JOIN 地狱                                  
  - 一个人可以同时是老师和家长，不需要两个账号                             
  - 登录认证只查一张表                        
  - 角色切换是前端行为，不是数据库行为

  三、权限模型（RBAC + 资源级）
                                                                           
  ┌─────────────────────────────────────────────────────────────────┐
  │                         权限判定链路                              │    
  │                                                                 │
  │  请求进入 → 提取 user.roles → 匹配角色权限 → 检查资源归属 → 放行/拒绝│ 
  │                                                                 │      
  │  示例：老师A 查看 学生B 的详情                                     │   
  │  1. user.roles 包含 'teacher' → 匹配老师权限                      │    
  │  2. 老师权限允许查看"自己班级的学生"                                │  
  │  3. 检查：学生B 是否在老师A 的某个班级中                            │  
  │     → class_teachers(teacher_id=A) ∩ class_students(student_id=B)│     
  │  4. 有交集 → 放行                                                │     
  └─────────────────────────────────────────────────────────────────┘

  数据可见性规则                                                           
   
  老师看到的数据边界：                                                     
    SELECT * FROM students                                                 
    WHERE student_id IN (
      SELECT student_id FROM class_students                                
      WHERE class_id IN (                                                  
        SELECT class_id FROM class_teachers WHERE teacher_id = 当前老师
      )                                                                    
    )                                                           
                                                                           
  家长看到的数据边界：
    SELECT * FROM students                                                 
    WHERE student_id IN (                                       
      SELECT student_id FROM parent_student_relations
      WHERE parent_id = 当前家长 AND binding_status = 'active'
    )                                                                      
   
  学生看到的数据边界：                                                     
    WHERE user_id = 当前学生（只看自己）

  场景 3：老师同时是家长                                                   
                                                                
  同一个 user 记录：
    { id: 'U001', roles: ['teacher', 'parent'], school_id: 'S001' }        
                                                                           
  对应两个 profile：                                                       
    teacher_profiles: { user_id: 'U001', title: 'IELTS老师' }              
    parent_profiles:  { user_id: 'U001', relation_type: 'mother' }         
                                                                           
  关系：
    class_teachers:           { teacher_id: 'U001', class_id: 'C001' }  ←  
  教 Year12 A班

  U008 是她孩子

  五、分层体系设计                                              
                                                                           
  学生分层不是固定的层级结构，而是标签化的动态系统：            
                                                                           
  student_profiles:
    tier: "intermediate"                    ← 主分层（简单分类）           
    tier_labels: {                          ← 详细标签（AI 生成 +          
  老师可调）                                                               
      "type": "目标型",                      ← 学习类型                    
      "focus": "Speaking 回避",              ← 当前重点                    
      "weakness": "Writing Task2 论证不足",  ← 薄弱项                      
      "style": "需要明确指令"                ← 学习风格                    
    }                                                                      
    risk_labels: {                          ← 风险标签（系统自动更新）     
      "consecutive_miss": 3,                ← 连续未完成次数               
      "speaking_avoidance": true,           ← 口语回避                     
      "exam_proximity": "sprint",           ← 临考状态                     
      "last_risk_at": "2026-03-28"                                         
    }                                                                      
                                                                           
  老师可以：                                                               
    ─ 查看系统生成的分层建议                                    
    ─ 手动调整 tier                                                        
    ─ 手动修改 tier_labels
    ─ risk_labels 由系统自动维护，老师只读                                 
                                                                           
  班级内分组：                                                             
    class_students.group_name = "冲刺组" / "A组" / NULL                    
    ─ 老师可按组布置任务                                                   
    ─ 组和分层是两个维度：分层描述能力，分组描述管理单元
